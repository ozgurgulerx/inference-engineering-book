# Contributing

Use small pull requests. A good PR changes one chapter section, one lab, one
diagram, or one checklist.

## Chapter PR Checklist

- The chapter has one clear reader outcome.
- Claims are backed by examples, measurements, or citations.
- New terms are added to `appendices/glossary.qmd`.
- Code examples are runnable or explicitly marked as pseudocode.
- Diagrams are text-based when possible.

## Review Questions

- Is the inference path clear?
- Are tradeoffs stated precisely?
- Could an engineer apply the chapter to a production system?
- Did the change introduce stale or vendor-specific assumptions?
